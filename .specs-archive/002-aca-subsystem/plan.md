# Implementation Plan: ACA Subsystem

**Branch**: `002-aca-subsystem` | **Date**: 2026-06-10 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/002-aca-subsystem/spec.md`

## Summary

Build an Autonomous Computer Agent (ACA) subsystem within JARVIS that transforms it from a voice-only assistant into a goal-driven desktop agent. The ACA observes the desktop via screen_vision, plans multi-step tasks using the existing LLM provider, executes actions through the tool registry, reflects on results for self-correction, and surfaces progress in the JARVIS UI. The subsystem lives in `core/agent/` with 8 modules and integrates with all existing action modules.

## Technical Context

**Language/Version**: Python 3.10+ (3.13 preferred) — matching existing JARVIS codebase

**Primary Dependencies**:
- Existing: `google-genai` (Gemini Vision for screen observation), `psutil` (process monitoring), `PIL/Pillow` (image processing)
- New: `pyautogui` (cross-platform mouse/keyboard control — lightweight, pure Python, no native compilation)
- No new external AI/LLM dependencies — reuses existing provider abstraction

**Storage**: JSON-backed via existing `memory/memory_manager.py` — goals, plans, execution history, and workflow templates serialized to `memory/agent_memory.json`

**Testing**: `unittest` — following existing conventions. New test class `tests/test_aca_subsystem.py` covering planner, executor, observer, reflection, approval_manager, and integration tests.

**Target Platform**: Desktop — Windows 10/11 (primary), Linux/macOS (secondary via pyautogui cross-platform support)

**Project Type**: Desktop application subsystem (in-process module within existing Tkinter app)

**Performance Goals**:
- Step execution: observe → plan → act → reflect under 15s per step
- Planning decomposition: under 10s for goals with up to 10 steps
- UI updates appear within 500ms of state change
- Agent overhead (vs manual tool use) under 20% additional latency

**Constraints**:
- Must not block JARVIS voice pipeline during agent execution (agent runs in background thread)
- Maximum 20 steps per goal, 5 min wall-clock limit per session
- Must respect `_user_initiated` security gate
- All shell commands pass through existing `actions/shell.py` security filters
- Human approval mode before high-risk actions (delete, kill, shell, download, service control)

**Scale/Scope**: Single-user desktop agent. No multi-agent, no server, no concurrent sessions in v1.
